# B 树(存储结构)
> ==请不要和算法的 B 树搞混==

#M阶 ：指的是 B 树中一个节点所含有的最大子节点数量。
> 假设 B 树中有一个节点含有 M(指的是最大值) 个子节点，则称这个 B 树为 M 阶 B 树

#根节点 ：指的是B 树中最顶层的那个节点，是整个树的入口。==但若根节点不是整个树的唯一节点，则强制要求根节点下必须分裂出两个分支==，防止退化成链表

#内部节点 ：指的是拥有父节点和子节点的节点，叫做内部节点
==若内部节点的数量为 M，则需要满足 (m/2)<=  M <=m 关系式==，(m/2) 向上取整。
你可以理解为，若一个内部节点内部节点只有一个，则该分支局部会退化成链表

#叶子节点 ：指的是没有子节点的节点，==需满足(m/2)-1<=K=m-1，其中 K 为叶子节点数量==
你可以理解为，若一个叶子节点内部只有一个，则该分支局部会退化成链表

<span style="background:#ff4d4f">B 树的所有的节点既保存键(key)又保存值 (value)</span>
## 插入
对于一个 m 阶高度 h 的 B 树遵循以下几个规则
- 若节点数量小于 m-1，则直接插入
- 若节点数量等于 m-1，则以节点中间的元素为分界，(偶数中间节点左右随机) 插入到父节点
- 重复以上两个操作，直到所有节点满足 B 树规则，最坏的情况也就是一直分裂到根节点，高度 h+1

## 删除
首先查找 B 树中需删除的元素, 如果该元素在 B 树中存在，则将该元素在其结点中进行删除
删除该元素后，首先判断该元素是否有左右孩子结点，
如果有，则上移孩子结点中的某相近元素 (“左孩子最右边的节点”或“右孩子最左边的节点”) 到父节点中.
移动判断如下：
- 某节点中的元素少于 ((m/2)-1), **其中 (m/2) 向上取整**，则需要看其同层相邻兄弟节点是否丰满
- 若丰满(既节点个数大于(m/2)-1)，则向父节点借一个元素满足条件。并且丰满的兄弟节点上移一个节点给父节点
- 若同层相邻兄弟节点不丰满，则兄弟节点与该节点进行合并，合并后仍需判断**内部节点**规则或者**叶子节点规则**

## 为什么要发明 B 树？
比如一个磁盘 7200 转，表示每分钟能转 7200 次，也就是说 1 秒钟能转 120 次，旋转延迟就是 1/120/2 = 4.17 ms；传输时间指的是从磁盘读出或将数据写入磁盘的时间，一般在零点几毫秒，相对于前两个时间可以忽略不计。那么访问一次磁盘的时间，即一次磁盘 IO 的时间约等于 5+4.17 = 9 ms 左右，听起来还挺不错的，一台 500 -MIPS 的机器每秒可以执行 5 亿条指令，因为指令依靠的是电的性质，换句话说执行一次 IO 的时间可以执行 40 万条指令，数据库动辄十万百万乃至千万级数据，每次 9 毫秒的时间，显然是个灾难。
考虑到磁盘 IO 是非常高昂的操作，计算机操作系统做了一些优化，当一次 IO 时，不光把当前磁盘地址的数据，而是把相邻的数据也都读取到内存缓冲区内，因为局部预读性原理告诉我们，当计算机访问一个地址的数据的时候，与其相邻的数据也会很快被访问到。每一次 IO 读取的数据我们称之为一页 (page)。具体一页有多大数据跟操作系统有关，一般为 4 k 或 8 k，也就是我们读取一页内的数据时候，实际上才发生了一次 IO，这个理论对于索引的数据结构设计非常有帮助。

## B 树的出现对什么会有极大的帮助？

B 树的每个节点的元素可以视为一次 I/O 读取，树的高度表示最多的 I/O 次数，在相同数量的总元素个数下，**每个节点的元素个数越多，高度越低，查询所需的 I/O 次数越少**；假设，一次硬盘一次 I/O 数据为 8 K，索引用 int (4 字节) 类型数据建立，理论上一个节点最多可以为 2000 个元素，2000 * 2000 * 2000=8000000000，80 亿条的数据只需 3 次 I/O（理论值），可想而知，B 树做为索引的查询效率有多高；

### 为什么开发人员要尽可能的降低 B 树的高度？

CPU 要去处理数据，那么就一定要到内存里读取，内存大小本来就远小于存储，那么就需要尽可能多地将存储的数据搬进内存中去。
但是磁盘存储运行时间作为计算机中最慢的几个硬件之一(这是物理层面上的限制，除非后续科技突破)，但是 B 树的每一层，都是需要发起一次 IO 查询的，因此 B 树降低高度，就是为了尽可能地减少 IO 地查询。
# B+树
## 定义
和 B 树的定义是类似的，但是 B+树有几个独特的属性
1. B+树的非叶子节点不存储值，只存储键(key)
2. B+树的叶子节点之间是互相连接的，叶子节点这一层构成了双向链表
3. B+树的叶子节点这一层存储了所有数据

## 特性
对于阶数为 m 的 B+树，它用有以下特性：
1. 每个节点最多拥有 m 个子节点
2. 每个节点最多拥有 m-1 个键值
3. 每个内部节点至少有 m/2 个子节点，根节点除外
4. 根节点至少有两个子节点
5. 1. 所有叶子节点出现在同一层级上，叶子结点与叶子结点之间有双向引用，叶子结点这一层构成了一条双向链表。
6. 一个具有 `k` 个子节点的非叶节点包含 `k-1` 个键值。
7. 叶子结点这一层存储了所有数据（所有的 key-link pairs）

# B+ 树适用场景
B+ 树非常适用于做文件系统索引和数据库索引，这是因为：
1. B+ 树的磁盘利用率更高
    - 这是因为 B+ 树内部节点仅存储 key，B 树内部节点既存储 key 又存储 value (link)，导致同样大小的磁盘空间可以容纳的 B+ 树内部节点要比 B 树多。这也就意味着如果使用 B+ 树索引，相同的查询需要的 IO 次数更少。
2. B+ 树支持高效的范围查询
    - 范围查询是非常常见的，因为 B+ 树叶子结点层构成了一个排好序的双向链表，因此对范围查询和整棵树的遍历是非常高效的。在 B 树结构中作范围查询的代价过高（类似在二叉搜索树中作范围查询，需要遍历整棵树），往往是无法接受的。
3. B+ 树的查询效率是稳定且可预期的
    - B+ 树的搜索最终会在叶子结点这一层结束，因为内部节点只存储 key 不存储 value (link)。所以每一次搜索的时间开销是相近且可预测的。

# 红黑树
[红黑树理解起来十分困难，可以参考这位大佬的文章](https://www.cnblogs.com/henuliulei/p/15114440.html)
## 规则
红黑树是一个<span style="background:#ff4d4f">自平衡但是不是绝对自平衡</span>的二叉查找树，树上的每一个节点都遵循以下规则
1. 每个节点要么红，要么黑
2.  但根节点始终是黑的
3. 叶子节点必须为黑色
4. 没有两个相邻的红色节点(不允许出现父节点是红色并且子节点也是红色)，但没说不能连续出现两个连续的黑色节点
5. 从任意节点(包括根节点)出发到后代 NUll 节点的每一条路径都具有相同的黑色节点
## 隐藏规则
1. 从根节点到叶子节点的最长路径不大于最短路径的两倍
	- 最短路径即为：从根节点到每个叶子节点的黑色节点数量是一样的，那么纯由黑色节点组成的路径就是最短路径
	- 最长路径即为：若有红色节点，则必然有一个连接的黑色节点，当红色节点和黑色节点数量相同时，就是最长路径，也就是黑色节点（或红色节点）* 2
2. 新加入的节点为红色节点
	- 当前红黑树中从根节点到每个叶子节点的黑色节点数量是一样的，此时假如加入新的黑色节点的话，则必然破坏规则，但加入红色节点却不一定，除非其父节点就是红色节点，因此加入红色节点，破坏规则的可能性小一些

## 删除需要考虑的情况
<span style="background:#ff4d4f">我们的目的是让要删除的值在红黑树中不存在即可</span>
红黑树删除的情况比较多，但也就存在以下情况:
1. 要删除根节点，则直接将根节点置为 NULL
2. 待删除节点的左右子节点均为 NULL，则直接将该节点置 NULL
3. 待删除节点的左右子节点有一个有值，则拿有值的替代待删除节点即可
4. 待删除节点的左右子节点都有值，则找前驱或者后继，将前驱或者后继的值复制到该节点，然后删除前驱或者后继(这里的前驱和后继需要来解释一下，前驱指的是子节点，后继指的是父节点)
